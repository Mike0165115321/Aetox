# Aetox CLI v1 — Target Outcome

Aetox CLI รุ่นแรกคือ coding agent ที่ทำงานจริงบนเครื่องผู้ใช้ แก้โค้ดจริง ตรวจผลจริง และใช้ `Aetox.md` เป็นแผนที่สถาปัตยกรรมเพื่อลด context waste โดยไม่ละทิ้งการตรวจสอบโค้ดจริง

## 1. Real Execution Loop

Aetox CLI ต้องมีลูปทำงานจริงที่รับคำสั่งผู้ใช้ วางแผน ใช้ tool/skill ทำงานบนเครื่อง รันคำสั่ง และคืนผลลัพธ์จริง ไม่ใช่เป็นเพียงระบบถามตอบ

ผลลัพธ์ที่ต้องเห็นได้:

* รับคำสั่งจากผู้ใช้
* วิเคราะห์ intent
* เลือกขั้นตอนทำงาน
* เรียกใช้ tool หรือ skill ที่จำเป็น รองรับการทำงานเป็นลูป
* ทำงานกับไฟล์และ command บนเครื่องจริง
* ส่งผลลัพธ์กลับให้ผู้ใช้ตรวจได้

## 2. Verified Coding Agent

Aetox CLI ต้องเป็น coding agent ที่เขียนโค้ด สร้างไฟล์ แก้ไฟล์ อ่าน error ตรวจผลลัพธ์ และรายงานสิ่งที่เปลี่ยนแปลงได้อย่างชัดเจน

ผลลัพธ์ที่ต้องเห็นได้:

* อ่านไฟล์ที่เกี่ยวข้อง
* สร้างไฟล์ใหม่เมื่อจำเป็น
* แก้ไฟล์เดิมอย่างมีเป้าหมาย
* รันคำสั่งตรวจสอบ เช่น test, build, lint หรือ command ที่เหมาะกับโปรเจกต์
* อ่าน error แล้วแก้ต่อได้
* รายงานว่าแก้อะไร ทำไมแก้ และผลตรวจเป็นอย่างไร

## 3. Architecture-Aware Context Governance

Aetox CLI ต้องจัดการ context อย่างฉลาด โดยใช้ `Aetox.md` เป็นแผนที่หลักของโปรเจกต์ ใช้หลักคิดแบบ senior architect เพื่อเข้าใจระบบ เลือกอ่านไฟล์เฉพาะจุด และลดการสร้างหนี้ทางสถาปัตยกรรมจากการแก้โค้ดแบบไร้วินัย

ผลลัพธ์ที่ต้องเห็นได้:

* เริ่มเข้าใจโปรเจกต์จาก `Aetox.md`
* ไม่อ่านทั้ง codebase แบบไร้เหตุผล
* เลือกอ่านไฟล์จริงเฉพาะส่วนที่เกี่ยวข้องกับงาน
* ตรวจว่า `Aetox.md` ยังสอดคล้องกับโค้ดจริงหรือไม่
* ไม่เชื่อเอกสารกลางแบบตาบอด
* ไม่แก้ architecture โดยไม่มีเหตุผล
* ลด context waste โดยยังรักษาความแม่นยำของงาน

## Target Workflow

ผู้ใช้สั่งงานโค้ด
→ Aetox เข้าใจโปรเจกต์จากแผนที่กลาง
→ เจาะไฟล์จริงเท่าที่จำเป็น
→ วางแผน
→ แก้โค้ด
→ ตรวจผล
→ รายงานกลับอย่างตรวจสอบได้

## Definition of Success

Aetox CLI v1 ถือว่าสำเร็จเมื่อผู้ใช้สามารถสั่งงานพัฒนาโปรเจกต์จริง แล้ว Aetox สามารถเข้าใจบริบท แก้โค้ด ตรวจผล และรายงานกลับได้ โดยไม่ผลาญ context เกินจำเป็น และไม่สร้างความเสียหายทางสถาปัตยกรรมแบบไม่รู้ตัว
