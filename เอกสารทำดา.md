## 📐 Minimal Viable Architecture (MVA) — "แค่นี้ก็ชนะแล้ว"

เพื่อไม่ให้โปรเจกต์ "ใหญ่เกินตัว" สำหรับคุณ (คนเดียว) ให้โฟกัสแค่ **4 เสาหลัก** นี้ก่อน:

```
┌─────────────────────────────────────┐
│  🏗️ AetoxClaw MVA v1.0              │
├─────────────────────────────────────┤
│  1️⃣ Intent Extractor                │
│     • แปลงคำสั่งผู้ใช้ → JSON Plan  │
│     • ใช้โมเดลเล็กได้ เพราะงานไม่ซับซ้อน │
│                                    │
│  2️⃣ Dispatcher + Retry Loop         │
│     • รันแผนแบบมี "โอกาสแก้ตัว"      │
│     • คือหัวใจของ "เล็กแต่ฉลาด"      │
│                                    │
│  3️⃣ WorkingMemory (2-layer พอ)      │
│     • RAM: เก็บสถานะปัจจุบัน        │
│     • Disk: บันทึกประวัติ + สรุป    │
│     • (ยังไม่ต้องมี Vector/Embedding)│
│                                    │
│  4️⃣ Tool Registry + Safety          │
│     • เพิ่ม/ลบ tool ได้ผ่าน YAML    │
│     • ตรวจสอบคำสั่งอันตรายก่อนรัน   │
└─────────────────────────────────────┘
```

### ✅ สิ่งที่ยัง "ไม่ต้องทำตอนนี้" (กัน Scope Creep)
| ฟีเจอร์ | เลื่อนไปเมื่อไหร่ | เหตุผล |
|---------|-----------------|--------|
| 🔹 Vector DB + Embedding | Phase 2 | ใช้ keyword search + summary ก่อนก็ได้ |
| 🔹 Multi-Agent (Planner/Critic แยก) | Phase 2 | ใช้โมเดลเดียวสลับบทบาทผ่าน prompt ก่อน |
| 🔹 GUI / Dashboard | Phase 3 | CLI + Discord Bot พอสำหรับทดสอบ |
| 🔹 Auto-Learning จาก Episode | Phase 3 | เขียน episodic memory แบบแมนนวลก่อน |
| 🔹 Distributed / Multi-GPU | Never (อาจ) | โฟกัสที่ "เครื่องเดียว 8GB" ให้สุดก่อน |

## 🚀 Scaling Strategy: "เล็กก็ได้ ใหญ่ก็ดี"

### 📊 ระบบตัดสินใจเลือกโมเดลอัตโนมัติ (Model Router)

```yaml
# config/models.yaml — เพิ่มส่วนนี้
model_router:
  # เกณฑ์เลือกโมเดลตามความซับซ้อนของงาน
  rules:
    - if: "task_complexity <= 3"  # งานง่าย
      use: "qwen3:4b"             # ใช้ตัวเล็กสุด → เร็ว
    - if: "task_complexity <= 7"  # งานปานกลาง
      use: "qwen3:8b"             # ใช้ตัวกลาง → สมดุล
    - if: "task_complexity > 7"   # งานยาก
      use: "qwen2.5:14b"          # ใช้ตัวใหญ่ → แม่นยำ
    - if: "user_override"         # ผู้ใช้บังคับ
      use: "{user_specified}"
  
  # fallback ถ้าโมเดลที่เลือกไม่พร้อม
  fallback: "qwen3:8b"
```

```python
# core/model_router.py
class ModelRouter:
    def select(self, task: Dict, user_override: str = None) -> str:
        if user_override:
            return user_override
        
        complexity = self._estimate_complexity(task)
        
        for rule in self.config["rules"]:
            if eval(rule["if"], {"task_complexity": complexity}):
                return rule["use"]
        
        return self.config["fallback"]
    
    def _estimate_complexity(self, task: Dict) -> int:
        """ประเมินความซับซ้อน 1-10 จากลักษณะงาน"""
        score = 0
        score += len(task.get("steps", [])) * 1.5      # หลายขั้น = ซับซ้อน
        score += 2 if "web" in task.get("tools", []) else 0  # ดึงเว็บ = ยาก
        score += 3 if "code" in task.get("output_type", "") else 0  # เขียนโค้ด = ยากมาก
        score += 1 if len(task.get("context", "")) > 2000 else 0  # บริบทยาว = ยาก
        return min(10, int(score))  # จำกัดที่ 10
```

### 🔄 ผลลัพธ์: ระบบ "ปรับตัวได้" อัตโนมัติ

```
ผู้ใช้: "ช่วยดึงข่าวจากเว็บข่าวไทย แล้วสรุปเป็นภาษาไทย"
          ↓
ModelRouter.estimate_complexity()
  • steps: 2 (ดึง + สรุป) → +3
  • tools: ["web_scraper"] → +2
  • output: "summary_th" → +1
  • total: 6/10 → ระดับปานกลาง
          ↓
เลือกโมเดล: "qwen3:8b" ✅ (สมดุลความเร็ว/ความแม่นยำ)

ผู้ใช้: "เขียนระบบ login ด้วย FastAPI + JWT + PostgreSQL"
          ↓
ModelRouter.estimate_complexity()
  • steps: 5+ (design → code → test → deploy) → +7.5
  • tools: ["code_executor", "file_manager"] → +0
  • output: "code" → +3
  • total: 10.5 → ปัดเป็น 10/10 → ยากมาก
          ↓
เลือกโมเดล: "qwen2.5:14b" ✅ (ใช้ความแม่นยำสูง)
```

> 💡 **จุดแข็ง:** ผู้ใช้ไม่ต้องคิดว่า "งานนี้ควรใช้โมเดลไหน" — ระบบเลือกให้เองตามความเหมาะสม
