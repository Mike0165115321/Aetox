## 2. Frontend (`desktop/frontend/src/`)

### 2.1 Layout: a 3-column, resizable CSS Grid

`App.svelte` defines the whole window as one grid: `sidebar | 6px handle | main (chat) | 6px handle | inspector (workbench)`. `main`'s column is `minmax(360px, 1fr)` — a hard floor enforced by the CSS grid track-sizing algorithm itself, independent of anything in JS.

Sidebar and inspector widths are user-resizable (drag the 6px handles), persisted to `localStorage`. Two real bugs were found and fixed here this session (full detail in [ARCHITECTURE.md §6.8](../../ARCHITECTURE.md#68-frontend-layout-bugs--fixed-2026-07-22), summarized):

1. **Stuck-drag / unbounded growth.** Dragging a handle across the native WebView2 browser window (§1, `browser.go`) let the OS deliver the drag-ending `pointerup` to that separate native window instead of back to the DOM — the drag state never cleared, and the panel kept growing on any later mouse movement. Fixed with `setPointerCapture` on the handle at drag start.
2. **No maximum width at all.** `clampSize` only enforced a floor (`Math.max(min, px)`) by original design, to avoid squeezing `main` below its own grid floor. Fixed by computing a real ceiling at drag time instead: `window.innerWidth − (other panel's current width) − 360 (main's floor) − 12 (two handles)` — same protection for `main`, without allowing runaway growth.

A third, unrelated CSS bug (blank-state text clipping instead of wrapping at narrow widths — `align-items:center` sizing text nodes to content width instead of the container) is also fixed, same ARCHITECTURE.md section.

### 2.2 `Workbench.svelte` / `BrowserPane.svelte` — bridging Svelte state to a native window

The workbench tab strip (`workbench.svelte.ts` store) is ordinary Svelte state — but the `browser` tab kind is not: its actual pixels come from the native WebView2 window in `browser.go`, not from anything in the DOM. `BrowserPane.svelte` is the bridge:

- On first URL, calls `BrowserOpen(tab.id, url, ...physRect(el))` — `physRect` converts the pane div's `getBoundingClientRect()` (CSS pixels) to physical pixels (`× window.devicePixelRatio`) for the Win32 `CreateWindowExW` call.
- A `ResizeObserver` on the pane div keeps the native window's bounds in sync (`BrowserSetBounds`) whenever the div resizes — including the resize-handle drag from §2.1, which is exactly why that bug was visible at all (Z-order made the tab visible; only then did its bounds tracking need to be correct).
- `BrowserSetVisible` hides the native window when its tab isn't active, or when the Settings overlay is open — because it's a real OS window, it would otherwise float above Svelte UI that's meant to be in front of it.

**Rule of thumb when debugging anything that looks like a browser-tab rendering glitch:** first determine whether the bug is in the Svelte/CSS layer (DOM, doesn't need Wails) or the native-window layer (`browser.go` + this bridge, needs an actual rebuild+run to observe — `svelte-check`/`go vet` can't catch native-window positioning or Z-order issues at all).

### 5.1 Typography

- System-wide font scale: every explicit `font-size` in `style.css` (64 declarations) was proportionally scaled ×16/13 (body base 13px → 16px), rounded to the nearest 0.5px to match the file's existing half-step convention. Mechanical, not hand-tuned — some fixed-size icon boxes (e.g. `.side-head .chev`, `.row .tw`, both `width:9-10px`) now hold slightly larger glyphs than their box; no `overflow:hidden` is set on them so this is a harmless minor overflow, not clipping.
- File-editor content font size is **deliberately not** part of that system scale — `.editor-ta` (the actual code/text content in `FileEditor.svelte`) reads `var(--editor-font-size, 15.5px)` instead, set by a new small store (`lib/editorFont.svelte.ts`, same shape as `theme.svelte.ts`: `$state` + `applyEditorFontSize()` + `initEditorFont()` called in `main.ts`, persisted to `localStorage`). User-facing control lives in Settings → Appearance (`Settings.svelte`), a number input, 10–24px range. Rationale: file content and UI chrome are different concerns (VS Code models this the same way — "Editor: Font Size" is separate from the rest of the UI's zoom).
- Same treatment for chat: `.chat` (message list) and `.composer .input` (compose box) read `var(--chat-font-size, 15.5px)` via `lib/chatFont.svelte.ts` (identical shape, 12–22px range), also exposed in Settings → Appearance.
- After user feedback that the +23% scale (§5) read too large, the system-wide scale (all 64 `font-size` declarations, i.e. everything *except* the two vars above, which didn't exist yet at that point) was reduced by a flat **−0.5px** across the board.
- **Reversed within the same session:** after several more rounds of "still too big, shrink this specific spot" requests, the user asked to reopen the system-font lock themselves rather than keep iterating through chat — self-service won over the earlier "lock it" decision. If a future session sees the lock note in old context and no control in `Settings.svelte`, trust the code: the control exists now (`lib/systemFont.svelte.ts`, exposed in Settings → Appearance as "ขนาดทั้งระบบ", 80–130%).
- That control is **not** a font-size var — style.css is fixed px everywhere (not em/rem), so a font-size var only cascades to elements with no explicit size of their own, i.e. almost nothing. Instead it's CSS `zoom` on `<body>` (`aetox-system-zoom` in localStorage), which scales the whole rendered subtree — text, icon glyphs, and their fixed-px containers together — as one unit. Legitimate here specifically because Aetox only ships on WebView2 (Chromium), where the non-standard `zoom` property is well supported.
- The Settings control shows/edits this as a **real px number** (e.g. "14.7"), not the raw zoom percentage — user found a percentage ungroundable ("can't aim for a specific number") after landing on 95% by trial and error and wanting to know what that actually meant in real terms. `lib/systemFont.svelte.ts` exports `SYSTEM_BASE_PX = 15.5` (must match `body`'s literal font-size in `style.css` — no automatic link between the two, keep them in sync by hand if either changes); `Settings.svelte` does `px = zoom * SYSTEM_BASE_PX` for display and the inverse on input. The stored/applied value is still the zoom multiplier — only the UI's number changed.
- Knock-on fix required for `zoom` to behave: `.app` used `height:100vh`, and `vh` doesn't resolve against the zoomed layout consistently in Chromium — changed to `height:100%`, which needed `#app` (the Svelte mount div in `index.html`, previously heightless) added to the `html,body,#app { height:100% }` rule so the percentage chain actually resolves.
- Not verified live in this session: whether the sidebar/inspector resize-handle drag math (`App.svelte`'s `clampSize`, reads `e.clientX`/`window.innerWidth`) stays correct at non-100% zoom. Chromium reports pointer coordinates in the same post-zoom CSS-pixel space as layout boxes, so it should, but this needs an actual drag test at e.g. 85% and 120% zoom before calling it solid.

### 5.2 i18n

User asked to build multi-language support into the frontend now, while the string count is still manageable, with an explicit target of 20+ languages eventually. Built as a minimal hand-rolled system — no dependency, because the actual need is "look up a string by key for the current locale," which a library adds ceremony around without adding capability at this scale.

- `lib/locales/th.ts` is the **source of truth for keys**: a flat `Record<string, string>` with dot-namespaced keys (`'topbar.openFolder'`, `'settings.mcpDesc'`, etc). `lib/locales/en.ts` is typed as `Record<keyof typeof th, string>` — add a key to `th.ts` without a matching entry in `en.ts` and the build fails. This is the entire safety net against silent missing-translation gaps; there is no runtime "missing key" warning.
- `lib/i18n.svelte.ts`: `i18n = $state<{ locale }>`, `t(key, vars?)` looks up the active locale (falls back to `th` on a bad key), `{var}` substitution via string replace, `setLocale`/`initLocale` persist to `localStorage` (`aetox-locale`). `initLocale()` runs in `main.ts` alongside the other init calls (theme, editor/chat font, system zoom).
- **All 12 frontend files migrated** in one pass (App/TopBar/Sidebar/Chat/TaskTimeline/Workbench+workbench.svelte.ts/ReviewPane/FilesPane/BrowserPane/ToolsPane/FileEditor/Settings/cockpit.svelte.ts) — grep for Thai characters (`[ก-๙]`) outside `lib/locales/` and `lib/i18n.svelte.ts` returns nothing, confirmed after the migration.
- **Footgun hit and fixed while migrating:** several components already used `t` as a loop variable (`{#each workbench.tabs as t}`) or a local variable name (`const t = activeTab`, `const t = Date.parse(iso)`), which silently shadows the imported `t()` translation function within that scope — Svelte/TS won't error on this, it just silently calls the wrong thing (or a `WorkbenchTab` object) instead of translating. Fixed by renaming every shadowing local (`t` → `tab`, `parsed`, etc.) in `Workbench.svelte`, `ReviewPane.svelte`, and `cockpit.svelte.ts`. **Anyone adding new code that both imports `t` and iterates over anything named `t` will reintroduce this** — there's no lint rule catching it, it's a naming convention to hold the line on.
- Settings → Appearance has a real language switcher (`localeNames` maps `'th'|'en'` → display name for the `<select>`), sitting alongside the other appearance-scoped controls (theme, system zoom, editor/chat font). Switching locale is instant and reactive for anything computed with `$derived` (Chat's starters/approvalOptions, Settings' sections, ToolsPane's groups) or evaluated directly in a template expression — Svelte 5's reactivity tracks the `i18n.locale` read inside `t()` either way.
- **Known staleness gap, accepted deliberately:** singleton workbench tab names (Review/Files/Tools) and dynamically-created ones (browser "New tab") are resolved to a string once, at tab-creation time, and stored as a plain `name: string` on `WorkbenchTab` — switching locale after a tab is already open does not retranslate its tab-strip label. Fixing this properly would mean storing a `nameKey` instead of a resolved `name` for the singleton kinds (browser/terminal/file tabs are inherently dynamic content and wouldn't benefit anyway) — not done, since it only matters for the rare mid-session locale switch with tabs already open. If this bites, that's the fix.
- Only two locales shipped (`th`, `en`) — building out the other ~18 languages the user mentioned is a translation-content task, not a structural one; the mechanism already supports it (add `locales/<code>.ts` + one line in `i18n.svelte.ts`'s `dictionaries`/`localeNames`).

### 5.3 Native file drag-and-drop

Dropping a file from Explorer onto the app window opens it as a main-content tab (same path as clicking it in the sidebar tree) — for handing the AI a file without hunting through the project tree.

- `main.go`: `options.App{ DragAndDrop: &options.DragAndDrop{ EnableFileDrop: true } }` — off by default in Wails, had to be turned on. Used `useDropTarget: false` in the `OnFileDrop` call (`App.svelte`'s `onMount`) so the *whole window* is a drop target, not just elements tagged with the `--wails-drop-target` CSS var — simplest, most discoverable default for a single-purpose desktop app.
- Wails' `OnFileDrop` gives real **absolute** OS paths (unlike plain browser `<input type=file>`/HTML5 DnD, which sandboxes `File.path` away) — that's specifically why this used the Wails runtime API instead of hand-rolling `dragover`/`drop` DOM listeners.
- Problem: every existing path-taking Go method (`ReadFile`, `WriteFile`, `safeSandboxPath`) expects a path **relative to the project's `SandboxRoot`**, joined via `filepath.Join(root, relPath)` — feeding it an absolute path directly would silently produce a broken joined path, not an error. Fixed by adding `app.go`'s `RelativizePath(absPath) (string, error)` (`filepath.Rel` against `SandboxRoot`, rejecting anything that resolves to `.` or escapes upward) — frontend calls this once per dropped path before handing the result to `openFile`. Dropping a file from outside the open project correctly errors and is silently skipped (not a partial/garbled open).
- Tests: `TestRelativizePathInsideProject`, `TestRelativizePathRejectsOutside` in `app_test.go`, same style as the existing `safeSandboxPath`/`ReadFile` tests.

### 5.4 FileEditor.svelte → Monaco Editor (real minimap)

User wanted a VS-Code-style minimap when viewing a file. A plain `<textarea>` (what `FileEditor.svelte` used before) structurally cannot have one — no access to rendered glyph layout. Offered a cheap hand-rolled canvas approximation vs. adopting Monaco Editor (VS Code's real editor component); user chose Monaco explicitly, accepting the dependency cost.

- **New dependency:** `monaco-editor` (^0.56.0). `FileEditor.svelte` now dynamically `import('monaco-editor')` inside `onMount` — not a static top-level import — so the ~4MB (uncompressed) editor core is a separate lazy chunk, not part of the app's initial load.
- `lib/monacoSetup.ts`: side-effect module (import once, ES module caching makes repeat imports from multiple open tabs free) that wires `self.MonacoEnvironment.getWorker` to real Web Workers via Vite's `?worker` suffix, for the editor/json/css/html/typescript workers. **Import path gotcha hit while wiring this:** monaco-editor 0.56's `package.json` `exports` map is `"./*": "./esm/vs/*.js"` — the `esm/vs/` prefix is *implied* by the map, so importing `monaco-editor/esm/vs/editor/editor.worker` (the path pattern most older tutorials use) double-prefixes and fails to resolve under Rollup/Vite's production build (passes in `vite dev`/`svelte-check`, silently — only `vite build` catches it). Correct form: `monaco-editor/editor/editor.worker?worker` (no `esm/vs/`).
- Also exports `detectLanguage(path)` — extension → Monaco language id, defaulting to `'plaintext'`. `.svelte`/`.vue` map to `'html'` as a rough approximation (Monaco has no dedicated Svelte grammar); this is a deliberate simplification, not a gap to "fix" without a real Svelte-in-Monaco grammar existing to swap in.
- **Bundle size, accepted as-is:** `import('monaco-editor')` pulls in Monaco's *entire* bundled language set (Swift, COBOL-ish ECL, Julia, Perl, Ruby, Solidity, …) via its default barrel export — around 4MB uncompressed, split into ~40 small per-language chunks by Rollup. Newer Monaco (0.56) restructured `basic-languages/<lang>/<lang>.contribution` (the old, documented, cherry-pickable pattern) into a single `basic-languages/monaco.contribution.js` barrel that imports internal, **undocumented** `languages/definitions/<lang>/register.js` modules — technically importable individually to trim the bundle, but relying on an undocumented internal path is a real fragility risk (could silently break on any monaco-editor version bump with no deprecation warning). Decided against that: Aetox is a **local desktop app** serving assets embedded in the Go binary via WebView2, not a webpage over a network — the usual "shrink the bundle for faster page load" motivation barely applies here (no network transfer; the cost is a few extra MB in the compiled `.exe` and a marginally slower one-time JS parse on first file open). If bundle size ever becomes an actual measured problem, that's the place to revisit — not preemptively now.
- Editor options wired reactively via `$effect`: `fontSize` follows `editorFont.size` (Settings → Appearance's file-content font control still works — Monaco doesn't read the CSS `--editor-font-size` var the old textarea used, so `editorFont.svelte.ts` no longer sets that var at all, just the JS-readable `editorFont.size`), theme follows `theme.name` (`'vs-dark'`/`'vs'`, Monaco's built-in themes — no custom theme defined against Aetox's semantic tokens, another acceptable simplification for now).
- Ctrl+S save is now a Monaco command (`editor.addCommand(monaco.KeyMod.CtrlCmd | monaco.KeyCode.KeyS, save)`) instead of a DOM `onkeydown` — required because Monaco owns keyboard focus internally once mounted; a wrapping `<div onkeydown>` would not reliably see the keystroke.
- `dirty`/`base`/`draft` state, the save button, and error display are all unchanged from the textarea version — `draft` is now kept in sync via `editor.onDidChangeModelContent`. Editor + model are disposed in `onDestroy` (each open file tab owns one long-lived `FileEditor` instance per `App.svelte`'s `{#each cockpit.openFiles as f (f.path)}` keying — confirmed no path-reuse-within-one-instance case to handle).

### 5.5 Code theme import (VS Code theme extensions)

Follow-up to 5.4: user noticed Monaco's `vs-dark`/`vs` aren't "our own" theme and asked to load real theme files, same as a VS Code theme extension. `lib/editorTheme.svelte.ts` is a separate concern from both the app's light/dark toggle (`theme.svelte.ts`) and file-content font size — same "independent, its own store" pattern used everywhere else in this session's Settings work.

- Settings → Appearance gained two rows: a code-theme `<select>` (built-in dark/light + whichever theme was last imported) and a "นำเข้าธีม (.json)" file picker (`<input type="file" accept=".json">` inside a styled `<label>` — no Go/Wails dialog needed, since reading a dropped-in JSON file's *content* client-side via the File API doesn't need a real filesystem path the way `ReadFile`/`WriteFile` do).
- **Core limitation, stated up front rather than discovered later:** VS Code theme JSON's `tokenColors` are TextMate scope selectors (`"keyword.control.flow"`, `"entity.name.function"`, …); Monaco's basic-languages tokenizers are Monarch grammars with their own, much smaller token vocabulary (`comment`, `string`, `keyword`, `type`, …). There is no exact mapping between the two systems — `vsCodeThemeToMonaco()` in `editorTheme.svelte.ts` does a best-effort prefix-regex match (`scopeToToken`) covering the common categories (comment/string/number/constant/keyword/function/type/tag/attribute.name/variable/delimiter/regexp/invalid) and drops anything it can't confidently map. **This means an imported theme will look "roughly right", not pixel-identical to the same theme in real VS Code** — expected and inherent to Monaco, not a bug to chase down. `colors` (the UI chrome — `editor.background`, `editor.foreground`, etc.) map through almost verbatim since Monaco reuses VS Code's color-id naming directly, so the overall background/foreground/selection colors of an imported theme should be accurate even where token colors aren't.
- Considered and rejected: a dedicated npm converter package (e.g. community TextMate→Monarch theme converters) — niche, uncertain maintenance, and the conversion problem is inherently lossy no matter whose code does it, so a purpose-built dependency wasn't clearly better than ~60 lines owned in-repo.
- Supports **one** imported theme at a time (`localStorage` keys `aetox-editor-theme-json` raw JSON + `aetox-editor-theme-choice`) — importing a new one replaces the old. A multi-theme library/manager was not built; nothing so far suggested the user wants to juggle several imported themes, and it's a small addition later if they do.

### 5.6 File-type icons — tried, then reverted to minimal

User's actual ask was broad — "support loading extensions: themes, icons, files" — not realistically scoped for VS Code's actual `.vsix` extension API (enormous). Narrowed step by step:

1. First built `lib/fileIcons.ts`: an extension→emoji map (🐹 for `.go`, 🔷 for `.ts`, etc.), replacing the flat "📄 for every file" the backend produces (`ProjectTree` in `app.go` only ever distinguishes dir vs file, `Icon: "📁"`/`"📄"` — never per-extension). Wired into `Sidebar.svelte` and `workbench/FilesPane.svelte`.
2. User then asked for the *real* Material Icon Theme specifically. Checked — it's published straight on npm (`material-icon-theme`, official package by the extension's own author, ~1.7MB, MIT) — a legitimate, low-effort path if pursued (no folder-picker/asset-unpacking machinery needed, unlike a generic "import any icon theme" system). **Not installed** — user interrupted before the `npm install` ran and, on reflection, decided against it as more than they wanted right now.
3. **Reverted to minimal**, explicitly requested ("VS Code without an icon theme installed, minimal") — `fileIcons.ts` deleted, both tree renderers back to a single generic 📄 for every file. The only icon differentiation left is folder open/closed (📂/📁), which was already there before this whole detour.

Net effect of this whole arc: **no change** from where §5.6 started, by the user's own choice — worth keeping this note so a future session doesn't rebuild the emoji-map or Material Icon Theme path without checking first; both were explicitly tried and turned down, not overlooked.

### 5.7 Tree clarity: depth-colored rows + a real chevron

Two small legibility fixes to the same file-tree rows (`Sidebar.svelte`, `workbench/FilesPane.svelte`), both user-reported after actually looking at the (by-then-reverted-to-minimal) tree:

- **Depth accent color.** Added `--depth-1`..`--depth-5` to `theme.css` (cycled 5-hue set — cyan/blue/violet/sky/amber, both `:root` and `[data-theme="light"]`). `lib/depthColor.ts`'s `depthTint(depth)` returns `var(--depth-N)` cycling by `node.depth % 5`, set as an inline `--depth-tint` custom property per row; `.row` in `style.css` uses `color-mix(in srgb, var(--depth-tint) 45%, transparent)` for a faint left border and `color-mix(... 7% ...)` for a barely-there background wash — "different nesting level reads as a different (faint) color" without it turning into a loud rainbow tree. `color-mix()` requires a modern Chromium (111+); fine here since Aetox only ships on WebView2.
- **Chevron, not a glyph.** The expand/collapse marker (`.row .tw`) was a Unicode `▾`/`▸` character — at the small tree font size it was reading as a plain dot, not a directional arrow, which is exactly the report ("shouldn't just be a dot, want a VS-Code-style arrow"). Replaced with a CSS border-triangle (`border-width:4px 0 4px 5px`, three transparent sides + one colored) that rotates 90° via `transform` on expand — a real drawn shape, not a font glyph, so it can't degrade into an unreadable dot at any font size or on any system font.

### 5.8 Depth-color: tried, tuned down, then removed

User's first look at §5.7 landed too strong ("hard to look at") and, comparing against real VS Code screenshots, pointed out the tree never actually highlights which file is currently open. Two different things, one turned out to be a real bug:

- **Tone-down, then full removal.** First cut the background wash (`color-mix(... 7% ...)`) entirely, keeping only a left-border accent at a lower mix (45%→22%). Still didn't land — user reported the per-row left border now read as each file being individually boxed/wrapped ("something enclosing every file"), not the flat plain-list look real VS Code actually has. Removed entirely: `border-left` gone from `.row`, `--depth-tint` inline style gone from both tree components, `lib/depthColor.ts` deleted, `--depth-1..5` tokens removed from `theme.css`.
- **The "boxed" look outlived the depth-color removal** — because it was never only from that. `.proj` (the tree's own wrapper `<div>`, shared by `Sidebar.svelte` and `workbench/FilesPane.svelte`) had its own pre-existing `border:1px solid var(--border-subtle); border-radius:9px; background:var(--surface-panel)` — a visually distinct panel/card look, unrelated to anything added this session, that made the whole tree read as "boxed" independent of per-row styling. Removed too (`.proj` is now layout-only: `display:flex; flex-direction:column; gap:1px`) — the tree now sits flush against the sidebar background, no border, no separate panel color, matching VS Code's actual flat tree.
- **Net result of §5.7 + this: back to a plain flat row list**, same as before either was tried — the one thing that survived the whole arc is the real CSS-triangle chevron (§5.7) and the active-file fix directly below, both independently confirmed wanted. Don't reintroduce per-depth row coloring or a `.proj` panel background without checking first; both were tried and explicitly rejected.
- **The active-file highlight was dead code, not just unstyled.** `cockpit.svelte.ts` had a whole `selectNode()` function that properly set `node.active` and called `openFile()` — but **nothing called it**. Both `Sidebar.svelte` and `workbench/FilesPane.svelte`'s actual click handlers call `openFileTab()` (the *workbench* store, opens the file as a tab in the right-hand inspector panel) instead — a completely separate tab system from `cockpit.openFiles`/`selectNode`'s main-content-tab path. `node.active` could never become `true` from a real click; the "active row" style had simply never fired. Fixed by computing it directly against the store that's actually driving the click: `class:active={workbench.activeId === 'file-' + node.path}` (matching `openFileTab`'s own `id: 'file-' + path` scheme) — no stored boolean, no dead second code path. Deleted `selectNode()` and `TreeNode.active` (types.ts) since nothing exercises that path; if a future session finds `cockpit.openFiles`/`openFile()` genuinely unused too, that's worth a second look, but wasn't touched here.
- `.row.active`: flat `--surface-selected` background, `font-weight:600`, `box-shadow:inset 2px 0 0 var(--accent)` — accent color deliberately matches `.tab.active`'s underline color (`style.css` ~line 175, shared `.tab` class used by both `App.svelte`'s main-content tabs and `Workbench.svelte`'s inspector tabs), since the user asked for the tree highlight and the open-file tab to "look consistent."

### 5.9 File-tree name font size — decoupled

Immediate follow-up: tree row text read as tied to the (locked) system zoom — correct, `.row` had no font-size of its own, just inherited `body`'s zoomed size. Given its own independent control, same shape as editor/chat font: `lib/treeFont.svelte.ts` (`--tree-font-size` CSS var, 11–18px, default 13px, `localStorage` `aetox-tree-font-size`), exposed in Settings → Appearance as "ขนาดฟอนต์ชื่อไฟล์". Applies to `.row` — shared by both `Sidebar.svelte`'s tree and `workbench/FilesPane.svelte`'s tree, one change covers both.

### 5.10 Settings pages: centered, not left-pinned

`.settings-content` was `flex:1; padding:48px 64px` with no centering — `.settings-card`'s own `max-width:760px` meant on any wide window the whole settings page (all four sub-pages: General/Appearance/Models/MCP) sat flush against the left padding edge with a large dead void on the right. Fixed by wrapping each section's content (the `h2`/`p.set-sub`/`.settings-card` siblings, previously direct children of `.settings-content`) in one new `.settings-inner` div, styled `max-width:760px; margin:0 auto` — centers the whole column in the available space instead of one card centering itself while its siblings (headings, descriptions) stayed left-pinned.
